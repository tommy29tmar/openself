# ADR-0014 — Chat-First Soul Proposal Review

**Status:** Accepted
**Date:** 2026-03-05
**Deciders:** Engineering

---

## Context

Soul change proposals are pending updates to a user's style profile (voice, tone, values, communication style). They are auto-generated by Circuit-A in `assembleBootstrapPayload` when the agent observes patterns in how the user communicates. Previously, no UI or mechanism existed to surface these proposals to the user — they accumulated in the DB and expired silently after 30 days.

The team identified this as a "hidden feature": the backend was fully operational, but users never saw or reviewed their proposals.

The question: how to surface soul proposals without adding another UI panel or notification system?

---

## Decision

Surface soul change proposals through natural conversation, not a dedicated UI panel.

1. **Detection:** A new `has_pending_soul_proposals` situation is detected post-Circuit-A in `assembleBootstrapPayload` — not in `detectSituations`. This is the single source of truth and captures proposals auto-created in the same bootstrap turn.

2. **Directive:** The `pendingSoulProposalsDirective` instructs the agent to bring up the proposal organically, never pressure the user, and let it go if they're uninterested.

3. **Tool:** A new `review_soul_proposal` tool accepts `proposalId` and `accept: boolean`, allowing the agent to commit the user's decision.

4. **`first_visit` eligible:** The `first_visit` state can persist for many turns (until `generate_page` is called). Excluding it would hide proposals for an extended and unpredictable period. The situation is therefore eligible in all states including `first_visit`.

5. **Guard removal:** The blanket `first_visit` guard in `getSituationDirectives` (which blocked all situation directives for that state) was removed. Eligibility is now declared per-situation via `eligibleStates` in `DIRECTIVE_POLICY`. Existing situations that excluded `first_visit` from their `eligibleStates` continue to be suppressed correctly.

---

## Consequences

**Positive:**
- No new UI component needed — the existing chat channel handles discovery and resolution.
- Same-turn proposals (auto-created by Circuit-A on the current turn) are immediately surfaced.
- Prompt injection risks are mitigated: all user/model-derived strings (overlay keys, values, reason, proposal id) pass through `sanitizeForPrompt` before reaching the system prompt.
- Overlay complexity is bounded: max 5 keys rendered, excess shown as "N more omitted".
- `DIRECTIVE_POLICY` is now the single truth for eligibility per situation — no blanket per-state guards.

**Trade-offs:**
- Users may not always see their proposal on the first eligible turn (the agent brings it up naturally, not as the very first message). This is intentional — it avoids jarring interruptions.
- If a user is blocked (`blocked` state), proposals are not surfaced. This is correct — a blocked user should not be distracted with style updates.

---

## Alternatives Considered

**Notification badge in the UI:** Rejected — adds UI complexity and violates the "conversation-first" principle.

**Dedicated settings panel:** Rejected — soul proposals are ephemeral and contextual; a panel implies permanence and discoverability that isn't warranted.

**Separate API endpoint for frontend polling:** Rejected — adds client-server coordination without benefit when the chat channel already has the user's full attention.

**Detect in `detectSituations` via opts:** Rejected — creates a split detection path (opts + production path in `assembleBootstrapPayload`), causing drift risk. Single detection path is simpler and correct.

## Implementation

- `src/lib/services/soul-service.ts` — `getPendingProposals` ordering fixed (FIFO: `asc(createdAt), asc(id)`)
- `src/lib/agent/tools.ts` — `review_soul_proposal` tool added after `propose_soul_change`
- `src/lib/agent/tool-filter.ts` — `review_soul_proposal` added to `ONBOARDING_TOOLS`
- `src/lib/agent/journey.ts` — `has_pending_soul_proposals` Situation type + `BootstrapPayload.pendingSoulProposals?` field + post-Circuit-A detection in `assembleBootstrapPayload`
- `src/lib/agent/policies/index.ts` — `SituationContext.pendingSoulProposals?` optional field
- `src/lib/agent/prompts.ts` — assembly wiring
- `src/lib/agent/policies/situations.ts` — `sanitizeForPrompt` + `pendingSoulProposalsDirective`
- `src/lib/agent/policies/directive-registry.ts` — blanket `first_visit` guard removed; new registry entry for `has_pending_soul_proposals`
- `tests/evals/soul-proposal-chat.test.ts` — 13 new tests (directive unit tests, production path integration, tool execute tests)
- `tests/evals/tool-filter.test.ts` — `review_soul_proposal` added to `ALL_TOOL_NAMES`
