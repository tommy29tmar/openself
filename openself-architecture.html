MEMORY TIER ANALYSIS: Valore reale per LLM (audit 2026-03-12)

Tier 1: Chat History (20 turns) CRITICAL

Budget: 22,000 token (88k chars) — Storage: messages table — Injection: Sempre presente nel context

Come funziona: Ultimi 20 turni (user+assistant+tool calls) caricati da assembleContext(). Walk backwards fino a saturare il budget. Minimo 2 messaggi sempre inclusi. Role whitelist filtering prima di inviare all'LLM.

Valore reale: Massimo. L'LLM vede la conversazione corrente, mantiene coerenza, capisce il contesto immediato. Senza questo tier l'agente sarebbe senza memoria a breve termine.

Tier 1b: Knowledge Base — Facts (fino a 120 facts) CRITICAL

Budget: 13,000-17,000 token (varia per state) — Storage: facts table — Injection: Sempre nel context (anche first_visit)

Come funziona: Facts immutabili al livello agent (delete+create per correzioni). DB layer usa upsert per connector re-sync. Ordinati da sortFactsForContext(): 5 più recenti garantiti + resto per rilevanza (confidence × recency × children count). Dedup via stableDeepEqual (key-order-independent JSON comparison). Formato: [category/key]: {value}. 5 connectors alimentano facts: GitHub (repos, skills, social), LinkedIn (experience, education, skills), RSS (articles), Spotify (artists, tracks, genres), Strava (sports, stats).

Valore reale: Core assoluto. Source of truth dell'identità utente. Tutta la generazione pagina, personalizzazione, e comprensione dell'agente parte dai facts. Budget alto e giustificato. Privacy: L'agente vede TUTTI i facts (inclusi private), il filtro filterPublishableFacts() si applica solo alla pagina.

Tier 1c: Soul Profiles MEDIUM-HIGH

Budget: 3,000-13,000 token (varia per state, runtime gate ≥10 facts per first_visit) — Storage: soulProfiles table — Injection: Condizionale: disabled per blocked, runtime-gated per first_visit

Come funziona: Versioned overlay JSON (voice, tone, values, selfDescription, communicationStyle). Compilato in prosa dall'agente. Richiede approvazione utente via propose_soul_change. Per first_visit: include: false ma budget: 3000 — attivabile runtime quando publishable facts ≥ 10.

Valore reale: Medio-alto per utenti attivi. Influenza tono/voce dell'agente + personalizzazione pagina (Stage 2 compiler). draft_ready ha il budget massimo (13k) per garantire personalità nella pagina.

Tier 2: Conversation Summaries MEDIUM-HIGH

Budget: 5,000-7,000 token — Storage: conversation_summaries table (cursor-CAS, 1 row/owner) — Injection: 4/6 states (esclusi first_visit e blocked)

Come funziona: Generato async dal worker (memory_summary job, triggered dopo ogni chat turn). Threshold: ≥5 messaggi non summarizzati. Rolling merge incrementale (non replace-all). CAS via compound cursor (cursor_created_at + cursor_message_id). Usa modello "standard" (Claude Sonnet), maxTokens: 800. Include buildJournalDigest() con tool usage. Prompt: third person, <500 words.

Valore reale: Medio-alto per utenti di ritorno. Comprime la storia conversazionale oltre i 20 turni. draft_ready include summary a 5k (budget ridotto vs 7k standard per lasciare spazio al soul 13k). Feed-forward: Il summary alimenta anche il prompt di session compaction come PREVIOUS CONTEXT.

Limiti noti: Nessuna validazione qualità output (solo empty-string check). message_count stores only current window count (non cumulativo). Nessun TTL/cleanup su righe vecchie.

Tier 3: Meta-Memories (Agent Memories) IMPROVED

Budget: 5,500 token — Storage: agent_memory table — Injection: 4/6 states (esclusi first_visit e blocked), top 15 scored

Come funziona: Due path di scrittura: (1) Agent via save_memory() tool (source="agent", cooldown 5/60s, confidence 1.0); (2) Worker via saveMemoryFromWorker() (source="worker", no cooldown, confidence 0.8, "pattern" type). Quota max 50 attive con eviction policy. Dedup via SHA-256 content hash (trim+lowercase). 4 tipi: observation, preference, insight, pattern.

Eviction Policy: Quando la quota 50 è raggiunta, evictLowestScoring() disattiva la memory con punteggio più basso — ma protegge un minimo di 5 agent memories (AGENT_FLOOR=5). Sostituisce il rigid quota enforcement. ADR-0015.

Scoring: getActiveMemoriesScored() — score = creationRecency × provenanceWeight × usageBoost:
• creationRecency = 0.5^(age_days/14) (half-life 14gg)
• provenanceWeight = {agent: 1.0, worker: 0.6}
• usageBoost: se referenziata = 0.5^(days_since_last_ref/28) (half-life 28gg); se mai referenziata = 0.5 (NEVER_REFERENCED_PENALTY)
Pre-fetch 50 più recenti, score, top 15 returned. Format nel context: - [type|category] content (e.g. - [observation|layout-preferences] User prefers compact sidebar).

Usage tracking: updateLastReferencedAt(ownerKey, memoryIds) aggiorna last_referenced_at per tutti gli ID memory che sopravvivono al context assembly. Chiamata async in onFinish del chat route. Paired ID tracking (Map<line, id>) attraverso tutte le fasi di truncation. Migration 0030 aggiunge colonna last_referenced_at TEXT. Defensive cap: chunked queries per rispettare il limite 999 parametri SQLite.

Memory API: GET /api/memory (lista scored active memories) + DELETE /api/memory/[id] (soft-deactivate, owner-scoped).

MEMORY SELF-MANAGEMENT policy: Blocco dedicato nel system prompt. Salva: preferenze stile/comunicazione, pattern ricorrenti. NON salvare: info fattuali (→ facts), istruzioni one-time, tool usage stats. CROSS-TIER AWARENESS: quando RECENT EVENTS e KNOWN FACTS sono correlati, menzionare la connessione naturalmente.

Worker extraction: Session compaction estrae fino a 3 patternsObserved per window (LLM fast tier, few-shot prompt, behavioral synthesis obbligatoria, ban mechanical stats). Light heartbeat estrae journal patterns (deterministic: repeated_tool, tool_sequence, correction_pattern).

Miglioramenti vs design precedente: Budget 3.5k → 5.5k. Top 10 → top 15. Worker path no-cooldown. Scoring 2-factor → 3-factor (+ usageBoost). Rigid quota → eviction con AGENT_FLOOR. Formato [memoryType] → [type|category]. Memory API + SELF-MANAGEMENT policy.

Limiti aperti: (1) saveMemoryFromWorker usa raw SQL (bypass Drizzle type safety); (2) Nessuna UI frontend per gestire memories (solo API REST).

Tier 4: Episodic Memory (Life Journal) IMPROVED

Budget: 3,000-5,000 token — Storage: episodic_events + FTS5 + episodic_pattern_proposals — Injection: PUSH automatica via getRecentEventsForContext()

Come funziona: Append-only event ledger con source provenance (chat, github, linkedin_zip, rss, spotify, strava). Smart context injection: 30-day window, per-source caps (chat: 10, connector: 3 each, total: 15). Block label: RECENT EVENTS (last 30 days, N events). Mutable block (partecipa al budget shrink loop).

FTS5 word-split: recall_episodes usa AND semantics per query multi-word: split via .split(/\s+/), ogni termine genera OR (category, key, value), termini combinati con AND. Stesso pattern di searchFacts.

Connector dual-write: GitHub (new repos + activity stream, first-sync guard via hasActivityBaseline), LinkedIn (certifications + articles, source "linkedin_zip"), RSS (new articles), Spotify (taste-shift detection ≥3/5 top-5 changed + stale fact archival dopo 3 sync assenti via staleSinceSync), Strava (workout + PR milestones). First-sync baseline rule: solo facts, zero events (seeds connector_items per dedup). ADR-0016.

Dream Cycle: consolidate_episodes worker: checkPatternThresholds (≥3 events/60d, recency 30d, source='chat' only) → evaluatePatternWithLLM (fast tier) → insertEpisodicProposal (30d TTL). Acceptance: atomic SQLite transaction (claim + write activity fact + logEvent). Rejection: 90-day cooldown.

Agent tools: record_event (insert + enqueue consolidation), recall_episodes (FTS5 word-split AND semantics, max 20), confirm_episodic_pattern (accept/reject proposals, recomposes draft).

Miglioramenti vs design precedente: Budget 0 → 3-5k. Pull-only → push automatica. Nessun connector event → 5 connectors dual-write. has_pending_episodic_patterns situation per surfacing. FTS word-split AND semantics. GitHub first-sync guard. LinkedIn source unificato a "linkedin_zip". Spotify stale fact archival.
