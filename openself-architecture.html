MEMORY TIER ANALYSIS: Valore reale per LLM

Tier 1: Chat History (20 turns) CRITICAL

Budget: 22,000 token — Storage: messages table — Injection: Sempre presente nel context

Come funziona: Ultimi 20 turni (user+assistant+tool calls) caricati da assembleContext(). Walk backwards fino a saturare il budget. Minimo 2 messaggi sempre inclusi.

Valore reale: Massimo. L'LLM vede la conversazione corrente, mantiene coerenza, capisce il contesto immediato. Senza questo tier l'agente sarebbe senza memoria a breve termine.

Tier 3A: Knowledge Base — Facts (fino a 120 facts) CRITICAL

Budget: 13,000-17,000 token — Storage: facts table — Injection: Sempre nel context (anche first_visit)

Come funziona: Facts immutabili (delete+create per correzioni). Ordinati da sortFactsForContext(): 5 più recenti garantiti + resto per rilevanza (confidence × recency × children count). Dedup via stableDeepEqual. Formato: [category/key]: {value}.

Valore reale: Core assoluto. È la source of truth dell'identità utente. Tutta la generazione pagina, personalizzazione, e comprensione dell'agente parte dai facts. Budget alto e giustificato.

Tier 3B: Soul Profiles MEDIUM

Budget: 8,500-13,000 token — Storage: soulProfiles table — Injection: Assente per first_visit e blocked

Come funziona: Versioned overlay JSON (voice, tone, values, selfDescription, communicationStyle). Compilato in prosa dall'agente. Richiede approvazione utente via propose_soul_change.

Valore reale: Medio-alto per utenti attivi. Influenza il tono e la voce dell'agente + personalizzazione pagina (Stage 2 compiler). Ma l'agente funziona anche senza soul — non è bloccante. Sottoutilizzato nella fase iniziale: per first_visit il budget è 0.

Tier 2: Conversation Summaries MEDIUM

Budget: 7,000 token — Storage: conversation_summaries table (CAS) — Injection: Solo returning_no_page, active_fresh, active_stale

Come funziona: Generato async dal worker (memory_summary job). Trigger: ≥5 messaggi non summarizzati. Cursor-based incrementale. CAS pattern anti-race. Usa modello "standard" (Claude Sonnet).

Valore reale: Medio-alto per utenti di ritorno. Comprime la storia conversazionale oltre i 20 turni in un sommario strutturato. Riduce la pressione sul Tier 1. Ma assente per first_visit e draft_ready — i nuovi utenti non ne beneficiano.

Tier 3C: Meta-Memories (Agent Memories) LOW

Budget: 3,500 token — Storage: agent_memory table — Injection: Solo returning, active_fresh, active_stale (top 10)

Come funziona: Richiede che l'agente chiami esplicitamente save_memory(). 4 tipi: observation, preference, insight, pattern. Cooldown 5 writes/60s. Quota max 50. Dedup via content hash.

Valore reale: BASSO. L'agente quasi mai chiama save_memory() spontaneamente. Top 10 su max 50 = pochissimo contenuto. Budget 3.5k irrisorio ma adeguato al poco contenuto. Problemi: (1) nessun prompt esplicito che spinga l'agente a salvare memories; (2) la distinzione observation/preference/insight non è chiara per l'LLM; (3) il valore aggiunto rispetto ai facts e summaries è marginale.

Tier 4: Episodic Memory (Life Journal) LOW

Budget: 0 token (zero) — Storage: episodic_events + FTS5 — Injection: MAI automatica, solo pull via recall_episodes()

Come funziona: Append-only event ledger. L'agente registra eventi via record_event(). Dream Cycle worker consolida pattern (≥3 eventi/60gg) e propone activity facts. has_pending_episodic_patterns situation segnala pattern pendenti. Ma i dati episodici non entrano mai nel system prompt.

Valore reale: BASSO. Sistema completamente "pull": l'agente deve ricordarsi di chiamare recall_episodes(), cosa che non fa spontaneamente. Il Dream Cycle funziona ma i suoi output (proposals) diventano facts (Tier 3) solo dopo approvazione. Problemi: (1) 0 token di budget = l'LLM non vede MAI gli episodi nel contesto; (2) nessun prompt che suggerisca all'agente di consultare gli episodi; (3) il valore aggiunto è indiretto (solo via Dream Cycle → facts).

DIAGNOSI E RACCOMANDAZIONI

Tier 3C (Meta-Memories): come renderlo utile

1. Auto-save nel prompt: Aggiungere istruzione esplicita nel TOOL_POLICY: "Dopo ogni sessione significativa, salva 1-2 observations su preferenze o pattern emersi".

2. Trigger nel session compaction: Il worker che compatta le sessioni potrebbe estrarre meta-memories automaticamente, senza dipendere dall'agente.

3. Ridurre la complessità: Unificare i 4 tipi (observation/preference/insight/pattern) in un solo tipo generico. L'LLM non distingue bene tra questi.

Tier 4 (Episodic): come renderlo utile

1. Budget dedicato: Allocare 2-3k token per un blocco "RECENT EPISODES" nel context di active_fresh/active_stale. Iniettare top 5 eventi più recenti.

2. Push, non pull: L'agente non dovrebbe dover chiamare recall_episodes() per vedere i propri dati. I dati dovrebbero essere già nel contesto.

3. Cross-reference con facts: Quando un episodio è rilevante per un fact esistente, annotarlo. "Last workout: 3 days ago (from episodic)" accanto al fact attività.

4. Il Dream Cycle funziona ma il suo valore è indiretto: il pattern diventa un fact solo dopo approvazione. Il percorso episodio → proposal → approvazione → fact → pagina è troppo lungo per essere percepito dall'utente.
